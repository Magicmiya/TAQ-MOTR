# TAQ-MOTR Configuration Guide

This document describes TAQ-MOTR's main configuration layout, command-line overrides, checkpoint configuration inheritance, and commonly used fields. See [QUICK_START.md](QUICK_START.md) for the complete workflow.

## 1. Configuration Design

The project uses one complete YAML file for each training recipe. The same main config supports:

- `MODE=train`: training or resuming training.
- `MODE=val`: validation inference followed by TrackEval evaluation.
- `MODE=test`: test inference followed by submission packaging.
- Single-GPU or multi-GPU execution, selected with `--gpus`, `--distributed`, and `torch.distributed.run`.

Separate single-GPU, multi-GPU, and evaluation YAML files are therefore no longer maintained for the same dataset.

The current main configs are:

| Training recipe | Config | Primary training data |
| --- | --- | --- |
| DanceTrack | `configs/dancetrack_train.yaml` | DanceTrack train |
| SportsMOT | `configs/sportsmot_train.yaml` | SportsMOT train |
| SportsMOT Mix | `configs/sportsmot_mix.yaml` | SportsMOT train+val and CrowdHuman train+val |
| MOT17 | `configs/mot17_train.yaml` | MOT17, CrowdHuman, and CityPersons |
| MOT20 | `configs/mot20_train.yaml` | MOT20 |

Configs with substantive model differences may still be stored separately. Do not create a new YAML when only the device count, run mode, or machine-specific path changes.

## 2. Paths and Runtime Fields

Environment-specific fields remain empty in the main configs:

```yaml
RESUME:
EVAL_FILE_PATH:

Dataset:
  dataset_root:
```

Pass dataset, checkpoint, and resume paths on the command line:

```bash
export DATASET_ROOT="/path/to/datasets/root"
export CHECKPOINT="outputs/DanceTrack_train/checkpoint_last.pth"

python main.py \
  -E DanceTrack_val \
  -M val \
  -C configs/dancetrack_train.yaml \
  -EFP "$CHECKPOINT" \
  -DR "$DATASET_ROOT" \
  --gpus 0
```

`PRETRAINED_MODEL` uses a repository-relative path:

```yaml
PRETRAINED_MODEL: weight/rtdetrv2_r50vd_6x_coco_ema.pth
```

It can be overridden with `-P/--pretrained_model`. A bare relative filename passed with `-P` is resolved from the repository's `weight/` directory first. Do not commit personal home directories, server mount points, or experiment-specific checkpoint paths to configs or public documentation.

## 3. Configuration Loading and Precedence

The precedence for ordinary runtime fields is:

```text
command line > startup YAML
```

Checkpoint evaluation adds model-structure recovery:

```text
runtime options
    |
startup main YAML
    |
checkpoint_root/train/config.yaml (strict model structure)
```

The effective loading order is:

1. Load the main YAML selected by `-C/--config_path`.
2. Override runtime fields with `-M`, `-DR`, `--gpus`, `-EFP`, and other command-line options.
3. Apply nested field combinations generated by `--sweep`.
4. Locate the checkpoint root when `MODE` is `val` or `test`.
5. Restore strict model-structure fields from `<checkpoint_root>/train/config.yaml` when it exists.
6. Preserve Decoder, life-cycle, and Inference fields that are allowed to change during inference.

The following top-level structures strictly follow the checkpoint's training config:

```text
MODULE_NAME
Backbone
Encoder
Decoder
Criterion
Query_updater
hidden_dim
num_classes
batch_norm
```

A limited set of Decoder inference fields may still be overridden by the startup config or a sweep:

```text
num_queries
hqg_init_feat_level
hqg_init_feat_levels
qpn_interact_max_state
hqg_num_learnable_memory
hqg_active_learnable_memory
hqg_num_blocks
hqg_det_dn_mask_track_memory
```

`Life_cycle_management.hidden_dim` and `num_classes` follow the training config. Other life-cycle fields may use the official inference settings from the current dataset's main config.

If no `train/config.yaml` exists next to a checkpoint, the main config still contains the complete model structure, so a released standalone `.pth` file can be evaluated directly. A custom architecture must use the main config that matches its checkpoint.

## 4. Command-Line Arguments

Arguments are defined in `configs/utils.py`. Common options are:

| Argument | Purpose |
| --- | --- |
| `-E`, `--exp_name` | Required experiment name. |
| `-C`, `--config_path` | Required YAML file, comma-separated YAML list, or YAML directory. |
| `-M`, `--mode` | `train`, `val`, or `test`. |
| `-O`, `--outputs_dir` | Output root directory. |
| `-P`, `--pretrained_model` | Initialization weights for training. |
| `-L`, `--checkpoint_level` | Gradient checkpoint level; `-1` selects it automatically. |
| `-R`, `--resume` | Training checkpoint to resume. |
| `-EFP`, `--eval_file_path` | A `.pth` file or a directory containing checkpoints to evaluate. |
| `-DR`, `--dataset_root` | Common dataset root. |
| `-V`, `--videos` | Comma-separated validation or test video subset. |
| `--gpus` | GPU ids, such as `0` or `0,1,2,3`. |
| `--distributed` | Enable distributed training or evaluation. |
| `--sweep` | Evaluate a Cartesian product of nested configuration values. |

Main configs default to `Available_gpus: 0` and `use_distributed: false`. Multi-GPU runs must use both `torch.distributed.run` and `--distributed`:

```bash
export GPUS="0,1,2,3"
export NPROC_PER_NODE=4

OMP_NUM_THREADS=1 python -m torch.distributed.run --nproc_per_node="$NPROC_PER_NODE" main.py \
  -E DanceTrack_train \
  -C configs/dancetrack_train.yaml \
  -DR "$DATASET_ROOT" \
  --gpus "$GPUS" \
  --distributed
```

`NPROC_PER_NODE` must equal the number of ids in `GPUS`.

## 5. Resolving `config_path`

Single config:

```bash
python main.py -E exp -C configs/dancetrack_train.yaml --gpus 0 -DR "$DATASET_ROOT"
```

A filename relative to `configs/` is also accepted:

```bash
python main.py -E exp -C dancetrack_train.yaml --gpus 0 -DR "$DATASET_ROOT"
```

Comma-separated configs:

```bash
python main.py -E compare -C configs/a.yaml,configs/b.yaml --gpus 0 -DR "$DATASET_ROOT"
```

Config directory:

```bash
python main.py -E batch_exp -C configs/<config_directory> --gpus 0 -DR "$DATASET_ROOT"
```

`.yaml` and `.yml` files in a directory run in filename order. Multi-config runs append each config filename to the experiment name to prevent output collisions.

## 6. Top-Level Fields

```yaml
EXP_NAME: DanceTrack_train
MODULE_NAME: TAQ_MOTR
CONFIG_PATH:
MODE: train
OUTPUTS_DIR: outputs/
NEW_OUT_DIR: true
PRETRAINED_MODEL: weight/rtdetrv2_r50vd_6x_coco_ema.pth
CHECKPOINT_LEVEL: 1
RESUME:
EVAL_FILE_PATH:

hidden_dim: 256
num_classes: 1
visualize: false
batch_norm: normal
```

| Field | Description |
| --- | --- |
| `EXP_NAME` | Default experiment name, overridden by the required `-E` argument. |
| `MODULE_NAME` | Model construction entry point, currently `TAQ_MOTR`. |
| `CONFIG_PATH` | Records the resolved config path at runtime; do not set it manually. |
| `MODE` | Defaults to `train`; use `-M` for validation or test inference. |
| `OUTPUTS_DIR` | Output root directory. |
| `NEW_OUT_DIR` | Whether training appends a timestamp to the output directory. |
| `PRETRAINED_MODEL` | Initialization weights for training. |
| `RESUME` | Training checkpoint to resume, normally supplied with `-R`. |
| `EVAL_FILE_PATH` | Checkpoint file or directory to evaluate, normally supplied with `-EFP`. |
| `visualize` | Enables online visualization. Disabled for standard training and inference. |

## 7. Training and Stage Policy

```yaml
Training:
  Available_gpus: 0
  use_distributed: false
  eval_after_epoch: 1
  seed: 42
  device: cuda
  epochs: 30
  lr_rate:
    backbone: 2.0e-5
    sampling_offsets: 1.0e-5
    query_updater: 2.0e-4
    default: 2.0e-4
  weight_decay: 0.0005
  multi_checkpoint: false
  resume_scheduler: true
  clip_max_norm: 0.1
  accumulation_steps: 1
```

`stage_policy` describes epoch-dependent clip lengths, batch sizes, sampling intervals, datasets, learning rates, and auxiliary supervision:

```yaml
Training:
  stage_policy:
    sample_steps: [0, 8, 14, 21, 24]
    stages:
      - sample_length: 2
        batch_size: 5
        sample_mode: random_interval
        sample_interval: 10
        use_aux_loss: true
        use_det_dn_aux: true
        use_dn: true
        high_conf_threshold: 0.5
      - sample_length: 3
        batch_size: 4
      - sample_length: 4
        batch_size: 3
      - sample_length: 5
        batch_size: 2
      - use_dn: false
```

Each stage only needs to declare changed fields. Undeclared fields inherit the previous stage's state. The number and boundaries of `sample_steps` and `stages` must agree with the training schedule.

## 8. Dataset

```yaml
Dataset:
  dataset_name: DanceTrack
  train_datasets:
    DanceTrack: []
  dataset_root:
  sample_length: 2
  batch_size: 5
  sample_mode: random_interval
  sample_interval: 10
  sampler_shuffle: true
  num_workers: 4
  persistent_workers: true
  prefetch_factor: 2
```

Main fields:

| Field | Description |
| --- | --- |
| `dataset_name` | Single target dataset used for validation and test inference. |
| `train_datasets` | Datasets and splits included in `MixedDataset` during training. |
| `dataset_root` | Common dataset root, left empty in main configs and set with `-DR`. |
| `eval_videos` | Optional video subset, overridden by `-V`. |
| `detector` | MOT17 currently uses the `FRCNN` sequences. |
| `sample_length`, `batch_size` | Initial training sampling values, normally updated by stages. |
| `num_workers`, `persistent_workers`, `prefetch_factor` | DataLoader options. |

Directory layout:

```text
$DATASET_ROOT/
|- DanceTrack/
|- MOT17/
|- MOT20/
|- CrowdHuman/
|- CityPersons/
`- SportMOT/
   `- dataset/
      |- train/
      |- val/
      `- test/
```

SportsMOT configs do not define `dataset_dir`. The code derives `$DATASET_ROOT/SportMOT/dataset` from `Dataset.dataset_root`, so `-DR` must point to the common dataset root.

## 9. Model-Structure Fields

`Backbone`, `Encoder`, `Decoder`, `Criterion`, and `Query_updater` define checkpoint-dependent model structure. When evaluating a checkpoint trained in this repository, prefer its `train/config.yaml` to avoid structural mismatches caused by later changes to a main config.

Common Decoder fields:

| Field | Description |
| --- | --- |
| `det_query_mode` | `generative`, `learnable`, or `hybrid`. |
| `num_queries` | Number of primary detection queries. |
| `num_denoising` | Number of detection denoising queries. |
| `hqg_init_feat_levels` | Feature levels used by HQG; `-1` selects the last level. |
| `hqg_num_blocks` | Number of HQG interaction blocks. |
| `hqg_num_learnable_memory` | Learnable-memory parameter capacity. |
| `hqg_active_learnable_memory` | Number of active entries; `null` uses all available entries. |
| `qpn_interact_max_state` | Maximum track state allowed to participate in HQG. |
| `merge_det_track_layer` | Decoder layer where detection and track queries begin to merge. |
| `cross_attn_method` | `CUDA` uses the compiled deformable-attention operation. |

Do not change Backbone, Encoder, Criterion, or Query updater structure during inference unless the run is an explicit evaluation ablation.

## 10. Inference and Life-Cycle Management

Result export settings:

```yaml
Inference:
  result_score_thresh: 0.5
  result_min_area: 100
  result_only_active: true
```

Online track life-cycle settings:

```yaml
Life_cycle_management:
  high_conf_threshold: 0.5
  new_born_threshold: 0.9
  track_thresh: 0.5
  miss_tolerance: 30
  Sudden_death_threshold: 0.3
  det_recover_enable: true
  det_recover_max_time: 30
  det_recover_min_history: 0
```

These sections have different responsibilities:

- `Life_cycle_management.*` controls track birth, retention, loss, death, and recovery.
- `Inference.*` only filters results before tracking TXT files are written.

Each dataset's main config already contains the official inference values previously stored in its evaluation config. Use the matching main config to reproduce default results; use `--sweep` for parameter studies.

## 11. Visualizer

Official main configs default to:

```yaml
visualize: false

Visualizer:
  enabled: false
  hook:
    enabled: false
  time:
    enabled: false
```

Visualization increases GPU memory, CPU, and disk usage. Offline visualization scripts enable only the required task at runtime and receive environment-specific dataset, checkpoint, or tracker paths through their arguments. They do not depend on a dedicated evaluation YAML.

## 12. Sweep

`--sweep` supports nested fields and Cartesian products of multiple values:

```bash
python main.py \
  -E DanceTrack_scan \
  -M val \
  -C configs/dancetrack_train.yaml \
  -EFP "$CHECKPOINT" \
  -DR "$DATASET_ROOT" \
  --gpus 0 \
  --sweep Decoder.hqg_num_learnable_memory=16,32 Life_cycle_management.track_thresh=0.45,0.5
```

Common experiment-name abbreviations:

| Field | Abbreviation |
| --- | --- |
| `Decoder.hqg_num_learnable_memory` | `LM` |
| `Decoder.hqg_active_learnable_memory` | `ALM` |
| `Decoder.hqg_num_blocks` | `HB` |
| `Decoder.qpn_interact_max_state` | `QIS` |
| `Decoder.num_queries` | `NQ` |
| `Life_cycle_management.new_born_threshold` | `NB` |
| `Life_cycle_management.track_thresh` | `TT` |
| `Life_cycle_management.Sudden_death_threshold` | `SD` |
| `Inference.result_score_thresh` | `RS` |

The following runtime fields cannot be changed by a sweep:

```text
EXP_NAME
CONFIG_PATH
MODE
OUTPUTS_DIR
NEW_OUT_DIR
Training.Available_gpus
Training.use_distributed
Dataset.dataset_root
```

Use the dedicated `-E`, `-C`, `-M`, `-O`, `--gpus`, `--distributed`, and `-DR` options for these fields.

## 13. Output Directories

Training:

```text
<OUTPUTS_DIR>/<EXP_NAME>[_timestamp]/
|- train/config.yaml
|- train/log.txt
|- checkpoint_last.pth
`- checkpoint_best_*.pth
```

Validation:

```text
<checkpoint_root>/val/inference_val/<EXP_NAME>_<checkpoint_name>/
```

Test:

```text
<checkpoint_root>/test/inference_test/<EXP_NAME>_<checkpoint_name>/
<checkpoint_root>/test/inference_test/submission_<EXP_NAME>_<checkpoint_name>.zip
```

Evaluation outputs are always written under the checkpoint root rather than the regular training output directory from the startup config.

## 14. Recommended Practices

- Use command-line options when changing the dataset root, GPUs, mode, checkpoint, or resume file. Do not duplicate a YAML for these changes.
- Update the corresponding main config for official training-recipe changes, and record the training data and stage semantics.
- Create an ablation YAML only when the model structure or experimental hypothesis changes substantially.
- Do not modify `train/config.yaml` inside a checkpoint directory; it is the reproducibility record for that checkpoint.
- Run a single-GPU smoke evaluation with `-V` before a full multi-GPU run.
- When releasing a standalone checkpoint, keep a complete matching main config in the repository. When releasing a full experiment directory, also retain its `train/config.yaml`.
